# 产品范围文档

## 1. 项目背景

本项目用于给 `ChatGPT Business` 提供一层自有身份入口。当前实现模型不是传统账号密码，而是：

- 管理员生成卡密
- 用户在 `ChatGPT Business` 输入完整邮箱后跳转到 `/sso`
- 用户在 `/sso` 输入 `卡密 + 邮箱前缀 + 邮箱后缀`
- 新卡首次使用时自动绑定邮箱
- 系统再以 `OIDC` 身份提供方的方式把用户带回 `OpenAI ChatGPT Business`

这样可以将“谁被发了卡”“谁绑定了哪个邮箱”“谁登录过 OpenAI”“谁被禁用”“谁导出了卡密”全部纳入自己的审计范围。

## 2. 已确认事实

基于用户提供的 OpenAI 官方文档，可以确认：

- `ChatGPT Team` 已更名为 `ChatGPT Business`
- `ChatGPT Business` 支持 `OIDC`
- `ChatGPT Business` SSO 配置是自助式的
- 启用 SSO 前必须先完成域名验证
- `ChatGPT Business` 不支持 `SCIM`
- `ChatGPT Business` 的 SSO 只影响 `chatgpt.com`，不影响 `platform.openai.com`
- OpenAI 支持 `Custom OIDC`
- OpenAI 建议 SSO 配置期间保持 `Optional`

这些事实决定了本项目首版不做：

- API Platform 接入
- SCIM 自动预配
- 多协议并行支持
- 多租户组织管理

## 3. 首版目标

首版必须实现：

1. 管理员后台登录
2. 卡密批量生成
3. 卡密 CSV 导出
4. 卡密状态管理
5. 用户通过 `/sso` 输入卡密和邮箱前缀登录
6. 新卡首次使用时自动绑定公司邮箱
7. 已绑定卡密的用户可直接重复登录
8. 系统对 `ChatGPT Business` 提供 OIDC 登录
9. 后台记录关键审计日志
10. 文档完整，便于运维和用户上手

## 4. 非目标

首版明确不做：

1. `platform.openai.com` SSO
2. `SCIM` 用户自动开通与回收
3. 多个 OpenAI 工作区复用同一套配置
4. 多客户端管理后台
5. 社交登录
6. 手机验证码登录
7. 邮件验证码登录
8. 自助找回密码邮件流程
9. 完整 RBAC 权限平台
10. 卡密长期作为主登录凭证

## 5. 核心角色

### 5.1 全局管理员

负责：

- 系统初始化
- OpenAI SSO 对接
- 生成卡密
- 导出卡密
- 邀请成员进入 ChatGPT Business 工作区
- 吊销卡密
- 禁用用户账号
- 查看审计日志

### 5.2 普通管理员

负责：

- 批量发卡
- 导出卡密
- 协助用户处理卡密失败、邮箱不匹配和登录失败

### 5.3 用户

负责：

- 使用卡密完成首次绑定
- 在 `/sso` 输入邮箱前缀和卡密
- 通过 OIDC 登录 `ChatGPT Business`

## 6. 业务规则

### 6.1 卡密规则

- 一张卡密只能首次绑定一次
- 一张卡密只能绑定一个用户账户
- 卡密可以配置有效期
- 卡密可以被管理员提前吊销
- 数据库不保存明文卡密，只保存哈希值
- 明文卡密只在生成导出当次可见

### 6.2 账户规则

- 用户不能自助公开注册
- 用户必须先有卡密才能进入 `/sso`
- 用户必须使用允许域名下的邮箱后缀
- 新卡第一次使用时自动绑定邮箱
- 已绑定卡密后，后续继续通过同一张卡登录
- 管理员可以吊销卡密或禁用用户

### 6.3 OpenAI 接入规则

- 首版仅接入 `ChatGPT Business`
- 首版仅支持 `OIDC Authorization Code Flow`
- 强制要求 `PKCE S256`
- 不支持 `Implicit Flow`
- 不支持 `Password Grant`
- `redirect_uri` 必须精确匹配
- 登录邮箱必须与 OpenAI 工作区中用户使用的邮箱保持一致

### 6.4 审计规则

以下操作必须记录审计日志：

- 管理员登录成功和失败
- 卡密生成
- 卡密导出
- 卡密吊销
- 卡密首次绑定成功和失败
- 卡密登录成功和失败
- OpenAI 授权成功和失败
- 账号禁用和启用

## 7. 关键流程

### 7.1 发卡流程

1. 管理员登录后台
2. 选择批量生成卡密数量、有效期、备注
3. 系统生成随机卡密并只在导出时提供明文
4. 管理员导出 CSV
5. 管理员将卡密通过公司内部渠道发给用户

### 7.2 `/sso` 登录流程

1. 用户在 `ChatGPT Business` 输入完整邮箱
2. OpenAI 按 OIDC 规则跳转到本系统 `/sso`
3. 用户输入卡密和邮箱前缀
4. 如果是新卡，系统自动绑定该邮箱
5. 如果是旧卡，系统校验该邮箱是否与卡密绑定一致
6. 系统记录审计日志并进入 OIDC 授权流程

### 7.3 OpenAI 登录流程

1. 用户在 `chatgpt.com` 输入公司邮箱
2. OpenAI 按已配置的 OIDC 规则跳转到本系统
3. 本系统要求用户在 `/sso` 完成卡密验证
4. 登录成功后签发授权码
5. OpenAI 到 `/token` 换取 token
6. OpenAI 验证 `id_token` 或读取 `userinfo`
7. 用户进入 `ChatGPT Business`

## 8. 成功标准

满足以下条件视为首版可上线联调：

1. 管理员可成功生成和导出卡密
2. 卡密只能首次绑定一次
3. 用户可通过 `/sso` 完成绑定并登录
4. OpenAI 可成功使用本系统完成 OIDC 登录
5. 关键数据不以明文敏感形式落库
6. 所有数据库写入均使用预处理语句
7. 上线所需文档完整

## 9. 验收标准

### 9.1 功能验收

- 卡密生成、导出、吊销可用
- `/sso` 单页登录流程可用
- 新卡首次绑定流程可用
- OpenAI SSO 联调成功
- 管理员可查看基本审计记录

### 9.2 安全验收

- 无明显 SQL 注入点
- 无开放重定向
- 登录后会话 ID 轮换
- OIDC 授权码一次性使用
- Token 有过期时间
- Cookie 启用安全属性

### 9.3 运维验收

- 支持 HTTPS 部署
- 配置项可外置
- 数据库初始化流程清晰
- 管理员有应急回退方案

## 10. 第二阶段可选扩展

1. 更细的卡密批次策略
2. TOTP 二次验证
3. SCIM 或内部 HR 同步
4. 多 OpenAI 工作区支持
5. API Platform SSO
6. 更细颗粒度管理员权限
